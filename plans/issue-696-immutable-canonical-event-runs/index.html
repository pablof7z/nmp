Immutable canonical event runs

Test whether packing byte-identical canonical event values into bounded immutable Redb runs removes per-event copy-on-write work without larger engine batches or query/recovery tradeoffs.

Very High Level

Epic #612 remains constrained by durable Redb work. Packed postings removed row-per-index-membership amplification, and larger engine cohorts reduce commit count but exceed the memory gate. The remaining per-event physical mutation is the canonical event value itself: NMP still inserts one Redb `EVENTS` row for every accepted event. Build a benchmark-only comparator that keeps the current event-ID table, observation rows, relay dictionary, packed postings, transaction size, durability, and event codec unchanged. The candidate concatenates each transaction's existing portable event values into one immutable canonical run with a checked offset directory. A tiny range catalog maps an event key to its run, then the offset directory returns the exact borrowed event bytes. No new parser, compression, truncation, or alternate NIP-01 representation is introduced. Prove exact event-key and ID lookup, borrowed matching, materialization, deletions through the existing death overlay, selective query results, truncation refusal, abrupt-exit atomicity, and exact reopen. Run at least 10 alternating fresh-process pairs against packed Redb and include mandatory compaction. Stop after the benchmark unless maintenance-inclusive store wall falls at least 20%, projected production improvement reaches 10%, writes do not increase, and RSS, stored bytes, reopen, and query p95 stay within 10%. A passing physical ceiling opens a separate governed-production issue; a failure reverts the adapter and merges only evidence.

Existing Rules Or ADRs

  • Issue #696 captures the consequence before implementation and links the work to epic #612.
  • The experiment preserves the existing portable event codec, exact event IDs, packed ordering, synchronous Redb transaction boundary, and facts-before-claims recovery rule.
  • No public noun, FFI/SDK surface, production schema, migration, or destructive API changes in this benchmark-only issue.

Rule Tension And Tightening

Possible loosening

  • No correctness or durability rule needs loosening. Dead canonical bytes remaining inside immutable runs count against stored-byte and maintenance gates rather than being hidden.

Possible tightening

  • Consider requiring future immutable-run formats to publish a checked offset-directory codec plus corruption and range-boundary falsifiers before performance qualification.

Alternatives Considered

  • Pack event IDs as well: deferred because retaining the exact point-lookup table isolates canonical-value row cost and avoids conflating lookup design.
  • Pack mutable relay observations in the first stage: deferred because provenance updates require a separate overlay/compaction policy; event values are immutable and provide the cheaper falsifier.
  • Append canonical bytes to a second file: rejected for stage 1 because it introduces a cross-file recovery protocol before proving the row-shape ceiling.
  • Increase transaction size again: #694 reduced commit cost but failed throughput and RSS gates.
