## Own discoveries (not in the feedback)

- **A. Coverage-forgery chain (escalates #1):** network-controlled filter contents (via `Derived`) + constructible FNV-64 collisions + `CoverageKey(as_u64)` as the durable watermark key ⇒ a crafted event set can forge `CompleteUpTo` for a filter never fetched ⇒ false authoritative-empty. The hash issue is not hygiene; it reaches ledger #7's core guarantee.
- **B. Ledger #5 self-erosion (escalates #4):** the ledger's mechanism text presupposes a verification step ("post-verification") that no code performs. By the ledger's own rule, that's a red build today.
- **C. known-gaps misdiagnosis:** the CPU-jank entry's own profiler evidence (RedbStore::query + serde_json top-of-stack) contradicts its "Swift-side work" framing; the entry should be split Rust/Swift.
- **D. Broken stated contract in `nmp-ffi/convert.rs`:** the module doc promises "never a panic" for foreign strings; the Literal→`to_nostr` path panics. The fix (item 3) restores the crate's own documented invariant.
- **E. #4 and #9 compound:** unverified ingest (no sig check) + no relay admission means a single forged kind:10002 event both *enters the store as the winner* and *directs new socket connections*. Fixing either halves the exposure; fixing #4 first is strictly better (it also protects the directory).
- **F. Coalesce merge precondition doesn't save 2b:** `a.limit == b.limit` (coalesce.rs:60) reads like a safety guard but merged same-limit filters still under-fetch. The one-line fix is to make the precondition `a.limit.is_none() && b.limit.is_none()`.
