Very High Level

Epic #612 still misses its initial-ingest gate, and the latest production evidence leaves the durable Redb transaction boundary as the largest remaining production owner. NMP currently admits at most 4,096 consecutive EVENT frames into one governed transaction. That limit transformed the original tiny-transaction cliff, but it was never compared with larger finite cohorts. First, use the existing production probe as a configuration-only falsifier. Compare 4,096, 8,192, and 16,384 event ceilings in fresh alternating processes while keeping the 8 MiB byte ceiling, 200 us coalescing wait, verifier shape, corpus, durability, and bounded queues fixed. Record actual bridge and store batch distributions, Redb commit time, packed publication and compaction, complete throughput, first row, RSS, allocations, writes, and exact reopen. Run both durable Redb and MemoryStore controls so a Redb gain cannot hide a serial-pipeline regression. Only the strongest passing cohort becomes a production candidate. Control frames and lifecycle events remain structural barriers, applied acknowledgements remain ordered, every cohort commits synchronously, and the existing byte ceiling remains unchanged. Focused bridge tests will prove the higher count bound, byte splitting, barriers, shutdown, and backpressure. Retain the change only if durable production throughput improves by at least 10%, MemoryStore does not regress more than 5%, RSS and first-row latency remain within 10%, duplicate replay stays above its epic gate, and a one-million run reopens exactly. Otherwise revert the candidate and merge only negative evidence.

Existing Rules Or ADRs

  • Issue #694 captures the consequence and links the work to epic #612 before implementation.
  • The candidate preserves bounded queues, the finite byte ceiling, control/lifecycle barriers, ordered applied acknowledgement, and one crash-atomic transaction per admitted cohort.
  • No public noun, FFI projection, persistence schema, destructive API, or migration changes; architecture gates 1 through 4 are unaffected and gates 5 through 6 still run.

Rule Tension And Tightening

Possible loosening

  • No rule or durability invariant needs loosening. Non-durable group commit is explicitly excluded even though #658 measured it as favorable.

Possible tightening

  • Consider requiring future changes to a production batching ceiling to report the observed batch distribution and adjacent-size controls, not only the configured maximum.

Alternatives Considered

  • Redb non-durable foreground commits plus a checkpoint: faster in #658 but can lose accepted foreground work and is not shippable.
  • Overlap commit with projection: #688 proved the mechanism but measured only a 2.3% durable gain and reverted it.
  • Migrate to LMDB or Fjall: the exact candidates failed RSS or wall/write/query gates in #658 and #691.
  • Increase the byte ceiling together with the count ceiling: rejected because it confounds the experiment and weakens the existing memory bound.
