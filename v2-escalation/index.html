The arc: two nouns, closed vocabularies, and the wallet fence

What these 15 decisions are, the through-line across them, and how the wallet scope question got escalated and ruled · ~5 min

TL;DR

  • 15 v2 architecture/scope questions were resolved in one session: 14 decided directly on doctrine, 1 (wallet, #219) escalated to the owner.
  • Every ruling holds the two-noun model intact — a query/observation and a write-intent/receipt — with zero new nouns admitted.
  • Type wins over convention: bug classes close only when a mechanism makes the wrong path unreachable, not when everyone remembers a rule.
  • Vocabularies (selectors, reducers, order keys, reactive inputs) stay closed and open only behind a real consumer falsifier — never speculatively.
  • Scope fences hold: no reducer language, no blessed engine inputs, and — per owner ruling — no wallet/payments surface through v2.

The through-line: almost every open question here reduced to "does this need a third noun, or is it a variation on the two we already have?" The answer was always no. Closed vocabularies over open-ended escape hatches; mechanisms that make the wrong thing unrepresentable instead of conventions to remember; scope fences that hold until a real application, not a hypothetical one, proves they need to move.

The 15 rulings

Collection grouping and ordering decided

Source: #5 + #9 — ruled jointly (same Collection-contract gate)

  • Grouped/aggregated collections (e.g. a DM conversation list) are expressible with a closed vocabulary — Window::Grouped { key: Selector, reduce: Reducer::Newest } — reusing the existing closed Selector vocabulary, no new noun.
  • The reducer set is closed and starts as exactly {Newest}; anything else needs its own consumer falsifier.
  • Grouping must be engine-owned once built: client-side grouping over a paged window is provably wrong (a peer's newest message can sit outside the window) — bug-class-ledger #13's failure mode, one level up.
  • Ordering: the closed OrderKey vocabulary survives. App-defined sort (e.g. interaction-frequency) is app-side presentation over engine-windowed acquisition, done in the view model — never an app comparator or mutable app-fed sort-key column inside engine ordering/cursor decisions.
  • Not v2-blocking: grouping builds only when a real consumer (a DM-list-shaped app) demands it; the Mixtape-shaped ordering probe remains the falsifier for revisiting ergonomics.

Why: docs/VISION.md:75-76 — app closures never enter demand, admission, routing, ordering, or cursor decisions. The Collection contract already landed as observe(query, window) (#474/#485), and ledger #13 is BUILT with the two-cursor resolution these issues proposed: engine owns one canonical (created_at DESC, event_id ASC) cursor; app presentation may reorder rows but never redefine cursor/group membership. Bug-class-ledger #3-5 keeps the reducer set closed and falsifier-gated — no reducer language, no per-app reducer registration.

Drafts module (NIP-37): spec only decided

Source: #13

  • Ratified as a spec placeholder; "do not build yet" stands, with two precise blocking dependencies.
  • nmp-nip37 will exclusively claim the NIP-37 draft kind(s) plus kind:10013 via KindClaim (ledger #22: routing only exists inside a claim).
  • Routing fails open to appRelay/fallbackRelay when no kind:10013 is declared — a draft is the user's own recoverable content, unlike NIP-17's fail-closed private-inbox class.
  • Write path (NIP-44-to-self) is ready; the read path is blocked on the decrypt-result feedback gap — no bespoke decrypt path is allowed around it.
  • No new nouns: drafts are ordinary canonical rows plus ordinary write intents.

Why: the load-bearing trust distinction is your-own-recoverable-content (fail open) vs. someone-else's-private-inbox (fail closed) — docs/design/routing-and-ownership.md:154. Ledger #6 (no widen operation on private/narrow routes) stays untouched because the draft lane is a recoverable-content class, not a private-inbox class.

Build gate: Unit E claim-table routing (ledger #22, PARTIAL under #521) and the M3-C decrypt-on-ingest feedback path must both land first.

NIP-module ownership and contextual composition decided

Source: #45

  • The issue's contract is confirmed as settled doctrine, now promoted into docs/VISION.md §4 (lines 192-214) and bug-class-ledger #14.
  • The one open design fork is ruled: kind ownership does not gate contextual route contribution. A module claims only its exact NIP-defined schema; a contextual operation may contribute closed typed tags/route facts to a foreign-owned immutable draft without claiming that draft's kind.
  • Canonical example ratified verbatim: nmp-nip51 exclusively owns kind:10009; nmp-nip29 composes that output into GroupRef/host context and claims neither 10009 nor 30002.
  • Next implementation unit: the NIP-29 contextual group-publication operation (bind GroupRef, add typed h/host-relay constraint, core signs once, ordinary receipt stream, no Swift-owned optimistic mirror).

Why: the legacy design "incorrectly gates every route contribution on kind ownership" (ledger #14, TARGET). No blessed kind:1/social-feed recipe ever lives in core (VISION §2.2, §4); AuthorOutbox-based composer workarounds are evidence of the gap, not an accepted pattern.

Bounded state delivery and overload/shortfall semantics decided

Source: #46

  • The bounded-delivery contract (docs/design/bounded-delivery.md §1-§4) is settled: every limit outcome is exactly one of semantics-preserving chunk/coalesce, cached rows + explicit shortfall, typed pre-acceptance rejection, or backpressure/disconnect with a diagnostic reason.
  • Ratified addition 1: GC-timed teardown (UniFFI Cleaner, any finalizer) is not a conforming teardown path on any platform — demand withdrawal must fire deterministically on explicit cancel/scope-close.
  • Ratified addition 2: relay-advertised NIP-11 limits (max_subscriptions, max_filters, max_limit, size limits) join the compiled plan's budget set — ignoring them while planning is the "decorative configuration" this issue forbids.
  • No new nouns, no aggregate health/overload score; numeric defaults stay implementation policy, but limit categories and observable consequences are architectural contract.

Why: ledger #2 (lost/leaked subscription) closed by deriving wire demand from refcounted descriptors — unbounded-latency refcount release reopens that class in time rather than in code. Ledger #17 (silent truncation) is PARTIAL; this closes the wire/relay-advertised-limit gap in it.

Typed reactive inputs beyond currentPubkey decided

Source: #48

  • Keep one reactive leaf: Reactive(CurrentPubkey) remains the only reactive grammar root. No new typed external input references are admitted now.
  • App-owned changing value sets are expressed through descriptor replacement (or literals / Derived / SetOp / a typed protocol query) — replacement already gives surgical, refcount-shared demand.
  • No blessed engine inputs like allAccountPubkeys, ever — that would move an app-owned fact into the engine.
  • A concrete, recorded admission test for reopening: a real consumer must show either that replacement breaks required handle/evidence continuity, or that it provably forces wire churn an engine-tracked dependency would avoid (measured, not asserted).

Why: zero falsifier has surfaced since 2026-07-11 across any consumer (29er, Falsifier, Gallery, demo) — under the consumer-inward rule (docs/design-record.md:146), speculative grammar is exactly what YAGNI-but-correct forbids. $currentPubkey is special because VISION names it as the one reactive root spanning app-owned identity and engine-owned derived routing (docs/VISION.md:89-91).

Context-scoped query acquisition evidence decided

Source: #49

  • The Derived-context invariant proposed in review is ratified: every Derived inner node must carry an explicit full contextual demand (or an equivalent closed value naming its own source authority and access context).
  • No implicit outer-context inheritance — an outer NIP-29 host/AUTH demand deriving from a NIP-51 list query must not route the inner query through the group host or borrow the outer AUTH evidence.
  • Default rule makes the invariant total: a fragment naming no context for an inner node gets ordinary/default authority and public access context, never the outer's — inheritance is unrepresentable, not just discouraged.
  • The rest of the issue (U1/U2 vocabulary signoff) was already decided; only implementation remains, tracked through #106/#107 and the AUTH half in #8.

Why: bug-class-ledger #18 is exactly this class — implicit inheritance would conflate contexts structurally, one level down where review can't see it. docs/VISION.md:56-59: NMP must never conflate acquisition evidence from different authorities or access contexts.

Typed state instead of thrown FFI errors decided

Source: #158

  • Typed state wins over thrown FFI errors, via a three-way split with landed precedent on each edge.
  • Runtime/environmental failure during observation (store degradation, executor saturation, transport/routing failure, shortfall) → typed state on the observation handle, never a throw.
  • Publish composition/acceptance failure → the first typed fact on the ordinary nonthrowing receipt stream (precedent: tampered pre-signed events surface as WriteStatus::Failed, never a synchronous FfiError).
  • Deterministic value invalidity at construction (malformed hex, invalid tags/signatures) stays a synchronous typed error — these are app programming bugs with no runtime state to represent.

Why: ledger #3-5 — a bug class closes only when a mechanism makes the bad path unreachable; hand-translating throws into view state at every call site is governance-by-convention. docs/VISION.md §2.3: a query delivers rows plus evidence, and failure evidence is evidence, not an exception.

NIP-77 eligibility, reconciliation, and fallback decided

Source: #205

  • (a) Eligibility is a typed, inspectable EligibilityDecision value per (descriptor, relay), carrying reasons — never an implicit boolean. Only a behaviorally-minted capability token admits NIP-77; NIP-11 advertisement alone can never mint it (ledger #8, BUILT).
  • (b) Strategy is reconcile-then-subscribe: NIP-77 covers the cached/backfill window; live REQ always covers ongoing arrival. One-shot per plan generation, never on a timer/polling.
  • (c) Honest state: reconciliation deltas enter only through the canonical store door; coverage watermarks update only from evidence actually obtained. Rejection/garbage/failure falls back to REQ plus a typed diagnostic fact recording why.

Why: doctrine already ordered "Negentropy FIRST, REQ second — DEFAULT not opt-in… only when probed" (docs/design-record.md). No global synced/completeness claim ever appears (VISION §2.3) — completion is a per-relay watermark fact.

Typed kind:0 profile reads and writes decided

Source: #208

  • kind:0 is the second instance of the pattern NIP-02 already landed: a guarded replaceable protocol operation (ledger #20) plus a typed live-winner projection over ordinary demand.
  • Two named write operations, both ledger-#20-guarded: editProfile (preserves every unowned field byte-honestly; stale base yields typed ReplaceableConflict) and createProfile (explicit no-base signup operation; guards on no existing canonical row, never a silent upsert).
  • Read side is ordinary demand plus a typed winner projection — no generic "resolved profile" object that becomes a competing semantic owner.
  • Exactly one module claims the kind:0 schema; profile UX, navigation, avatar policy stay entirely application-owned.

Why: docs/VISION.md §7 table — presentation never lives in core (ledger #12); the #75 precedent requires create-when-absent to be its own named operation, never folded silently into edit.

Search authority: entity resolution, NIP-50, local FTS decided

Source: #215

  • "Search" is not a noun. Three existing things stay separate: codecs (NIP-19/21 decoding, NIP-05 HTTP resolution via the NIP-11 precedent), NIP-50 (a filter field — Selection — under capability-scoped source authority), and (if ever built) local FTS as a store capability.
  • NIP-50 results are ordinary canonical rows plus per-source evidence — no completeness claim; demand no capable relay can serve is uncovered demand with explicit shortfall evidence, never silently widened or dropped.
  • Local FTS is an explicit non-goal for v2 core — no consumer falsifier demands it. Its shape-if-built is pre-ruled: a store-layer capability inside the same crash-atomic transaction as canonical mutations, local-authority-only.
  • Ranking, tabs, result layout, and any blended score are application policy, never a health/completeness number in core.

Why: docs/design-record.md:146 (consumer-inward rule); ledger #7/#8 (per-source honesty, advertisement ≠ proof) applied symmetrically to search.

Web-of-Trust fact and computation boundary decided

Source: #217

  • NMP exposes trust facts through ordinary live queries — now and always. No WoT computation ships in core, and no computation module is built until a consumer falsifier demands one.
  • Inputs (follow edges, mute lists, reports, relay lists) are ordinary Nostr facts via live queries/module projections — acquisition authority and trust policy never mix.
  • If ever built, the admissible shape is pre-ruled: a replaceable consumer above the engine (the nmp-content precedent) — deterministic, versioned, explicit typed inputs, resource-bounded, cancelable, structurally unable to open relays or filter a live query.
  • Thresholds, allow/hide decisions, ranking composition, and moderation meaning are app-owned, permanently. The no-NIP-85 boundary is retained.

Why: docs/VISION.md:75-76 / §2.3 — nothing app-semantic enters demand/admission/routing, and rows are never filtered by an engine verdict; ledger #7's honesty rule applied to derived trust values.

Bounded live-only observation and publication of ephemeral events decided

Source: #393

  • Ephemeral events (kinds 20000-29999) fit the existing two nouns with zero new nouns.
  • Read side: a typed, bounded, live-only delivery lane on the ordinary observation handle, explicitly distinct from the canonical row set. No active observer means no buffering and no replay (honest live-only semantics, not data loss).
  • Write side: the ordinary immutable-draft/signer/routing/receipt pipeline in its explicitly non-durable mode — receipts tell relay-acceptance truth, never claim cache presence.
  • Eligibility is a protocol fact driven by the kind range, not an app flag.

Why: a query snapshot is defined as current matching cache rows plus evidence (docs/VISION.md:105-112); injecting an ephemeral as a pretend row would let a conflated/windowed observer silently disagree with a delta observer's accumulated state. Ledger #13/#17/#2 govern the bounded-buffer and refcount mechanics.

Pre-signature write cancellation with typed durable evidence decided

Source: #533

  • Built as specified: Cancelled becomes a typed durable receipt fact, and the current Failed("write cancelled before signing") string-matched projection is deleted in the same change — no compatibility side door.
  • Exact outcome semantics by prior state: accepted-unsigned cancel commits atomically; already-cancelled is idempotent; unknown receipt id is a typed refusal; already-signed/terminal is a typed refusal naming the current state; persistence failure leaves the obligation live with no false terminal fact.
  • Surfaced as cancel(receiptId) on the canonical nmp::Engine facade plus governed UniFFI/Swift/Kotlin projections — the Destructive-API gate and Cross-SDK Parity gate both apply.

Why: docs/bug-class-ledger.md:3-5 — string-matching is convention, not mechanism. VISION §3.3: a durable intent remains live until explicit cancellation; VISION §3.1's one-way-door rule extends symmetrically to cancellation.

Wallet protocol ownership and relay receipts vs. outcomes owner escalation

Source: #219 (closed) — ruled by @pablof7z, 2026-07-16

  • Wallet is a non-goal through v2. NMP ships no wallet/payments surface: no zap helpers, no NWC/NIP-47 credentials, no NIP-60/61 wallet state.
  • Apps hand-roll payments; NMP stays out of payment-outcome correctness — a relay accepting an event is not payment success, and the engine adds zero new credential surface.
  • The doctrine-decidable constraints already worked out remain the pre-ruled shape if/when it reopens post-v2: the recommended first step then is NIP-57 zap-request composition only (typed drafts + receipt correlation, no wallet state), gated on a real consumer falsifier.
  • This is the one ruling that could not be settled on doctrine alone — the scope-fence call needed explicit owner sign-off rather than a doctrine derivation.

Status: owner-escalated and decided. Issue closed as decided; reopens post-v2 only on consumer demand.
